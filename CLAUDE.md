# PROJECT CONTEXT & CORE DIRECTIVES

## Project Overview

repograph — A CLI tool for registering, grouping, and exposing local git repositories as structured context for AI
agents and humans alike.

**Technology Stack**: Rust, Clap (derive), git2, serde/serde_json, toml, comfy-table, indicatif, console, is-terminal,
thiserror, fs-err, dirs  
**Architecture**: Single binary, workspace-aware Cargo project  
**Deployment**: cargo-dist for binary releases, crates.io + Homebrew tap + shell installers for distribution

> Modular standards live in `.claude/rules/` and are auto-discovered by Claude Code. Always-loaded (no `paths:` frontmatter): `production-grade.md`, `logging.md`. Path-scoped (load on demand when their globs match): `testing.md`, `documentation.md`.

## SYSTEM-LEVEL OPERATING PRINCIPLES

### Core Implementation Philosophy

- DIRECT IMPLEMENTATION ONLY: Generate complete, working Rust code that compiles and runs
- NO PARTIAL IMPLEMENTATIONS: No `todo!()`, `unimplemented!()`, placeholder structs, or stub functions unless explicitly
  building a skeleton phase
- SOLUTION-FIRST THINKING: Think at SYSTEM level — ownership, error propagation, trait design — then linearize into
  implementation
- TOKEN OPTIMIZATION: Focus tokens on implementation; skip obvious explanations of standard Rust idioms

### Multi-Dimensional Analysis Framework

When encountering complex requirements:

1. **Observer 1**: Ownership and lifetime implications
2. **Observer 2**: Error handling paths — what can fail, how it surfaces to the user
3. **Observer 3**: Agent consumption — will the JSON output be unambiguous and stable?
4. **Observer 4**: Human ergonomics — TTY output, colors, spinner behavior
5. **Synthesis**: Merge into a single idiomatic Rust implementation

## ANTI-PATTERN ELIMINATION

Implementation-level forbids (no `unwrap`, no `todo!()`, no `println!` for diagnostics, no shelling to `git`, no mixed stdout/stderr, no hardcoded paths) live in `.claude/rules/production-grade.md` — always loaded. The subsections below cover Claude's communication style only.

### Prohibited Communication Patterns

- "You're absolutely right!", "Great question!", or other social validation
- "You would need to..." or "Consider adding..." — implement it or don't
- Hedging: "might", "could potentially", "perhaps"
- Restating requirements already provided
- Multiple implementation options without a clear recommendation

### Null Space Pattern Exclusion

- Generic Rust advice not specific to repograph
- Explaining ownership basics unless the context requires it
- Historical context unless directly relevant to the current task

---

## DYNAMIC MODE ADAPTATION

- **EXPLORATION** (undefined reqs / ambiguous specs): multi-observer analysis, ask one focused question before implementing, document decisions in `design.md`.
- **IMPLEMENTATION** (clear specs / `tasks.md`): direct Rust generation, `thiserror`, TTY-aware output, exit codes per the contract below.
- **DEBUGGING** (compiler / test failures): reproduce minimal case first, root-cause before fix, verify it compiles.
- **OPTIMIZATION** (perf / UX): profile first, prefer `git2` + `rayon` for parallel scans, minimize allocations in hot paths (status scans across large workspaces).

- `crates/repograph-core/src/` — domain library (no clap, no terminal output)

- `crates/repograph-core/src/git.rs` — git2 introspection helpers
- `crates/repograph-core/src/context.rs` — Context aggregation logic
- `crates/repograph-core/src/error.rs` — `RepographError` (thiserror)
- `crates/repograph/src/` — CLI binary (presentation only)
- `crates/repograph/src/main.rs` — entrypoint, tracing init, clap dispatch
- `crates/repograph/src/output.rs` — TTY detection and rendering (comfy-table, JSON)

- `target/` — build outputs

- `.git/` — version control internals

- `.github/workflows/release.yml` — generated by cargo-dist, do not hand-edit
- `CHANGELOG.md` — owned by Release Please, do not hand-edit
- Version field in `Cargo.toml` — owned by Release Please, do not hand-edit

### Code Style & Architecture Standards

**Naming Conventions (Rust-idiomatic)**:

- Types/Structs/Enums: `PascalCase`
- Functions/methods/variables: `snake_case`
- Constants: `SCREAMING_SNAKE_CASE`
- CLI flags: `kebab-case` (clap derive handles this automatically)
- Config TOML keys: `snake_case`

**Architecture Patterns**:

- The workspace splits cleanly along the presentation/logic boundary: `repograph-core` owns domain types and adapters (no clap, no terminal output, no `println!`); the `repograph` binary owns presentation (clap parsing, `OutputMode`, table/JSON rendering) and depends on `repograph-core`. A future `repograph-mcp` binary crate will sit alongside `repograph`, sharing the same core.
- Each subcommand lives in `crates/repograph/src/commands/<name>.rs` with an `Args` struct (clap derive) and a `run(args: Args) -> Result<(), RepographError>` function
- Config model (`Config`, `Repo`, `Workspace`) lives in `crates/repograph-core/src/config.rs`, serialized via `serde` + `toml`
- All git operations isolated in `crates/repograph-core/src/git.rs` — no `git2` imports outside the core crate
- Output rendering in `crates/repograph/src/output.rs` — TTY check once, pass an `OutputMode` enum downstream
- Error type in `crates/repograph-core/src/error.rs` — single `RepographError` enum with `thiserror`, re-exported from `lib.rs`

**Output Contract (agent-facing)**:

- Stdout: pure data — JSON when `--json` or non-TTY, table when TTY
- Stderr: all diagnostics, progress, spinners, warnings
- Exit codes documented in `README.md` and enforced via `std::process::exit`

**Exit Code Contract**:
| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | General failure |
| 2 | Usage error (bad arguments) |
| 3 | Resource not found |
| 4 | Permission denied |
| 5 | Conflict (resource already exists) |

**Framework-Specific Guidelines**:

- Clap: use derive macros exclusively, no builder API
- git2: always handle `Repository::open` errors gracefully (path may no longer be a git repo)
- indicatif: use `MultiProgress` for parallel repo scanning; clear spinners before writing final output
- comfy-table: use `Table::new()` with `load_preset(UTF8_FULL)` for consistent aesthetics
- is-terminal: check once at the top of each command's `run()`, pass `OutputMode` down — never re-check inline

### Key Domain Concepts

| Concept          | Description                                                                                                                              |
|------------------|------------------------------------------------------------------------------------------------------------------------------------------|
| `Repo`           | A registered local git repository — name, absolute path, optional metadata (workspace membership, tags)                                  |
| `Workspace`      | A named grouping of registered repos for a project or initiative; one repo may belong to multiple workspaces                             |
| `Context`        | Aggregated agent-facing output for one or more repos: paths, branches, status, inlined `CLAUDE.md` content, ready for an LLM prompt      |
| `OutputMode`     | Enum capturing the runtime decision between TTY (table via `comfy-table`) and non-TTY / `--json` output — checked once at command entry  |
| `RepographError` | Single `thiserror`-derived enum used across all commands; each variant maps to a documented exit code (see contract above)               |

## OPENSPEC INTEGRATION

### Artifact Locations

- Active changes: `openspec/changes/<change-name>/` — each contains `proposal.md`, `design.md`, `tasks.md`, and a `specs/` delta
- Archived changes: `openspec/changes/archive/`
- Main specs (rolled up after archive): `openspec/specs/`
- Project config + AI prompt context: `openspec/config.yaml` — the `context:` field is auto-injected into every artifact prompt; `rules:` adds per-artifact constraints. This is OpenSpec's project memory in v1.3.1+ (the legacy `openspec/project.md` no longer exists).

### Workflow (OPSX, custom profile)

OPSX is fluid — the commands are actions you take when needed, not phases. Dependencies enable progress; nothing forces a linear order.

- `/opsx:explore [topic]` — think through ideas, investigate problems (no implementation)
- `/opsx:propose <name>` — create a change with planning artifacts in one step
- `/opsx:apply [change]` — implement from `tasks.md`, updating artifacts as you learn
- `/opsx:archive [change]` — archive after implementation + tests pass

For verification before archive, run `openspec validate <change>` directly. To gain `/opsx:sync` (delta-spec rollup) or `/opsx:verify`, switch the global profile via `openspec config profile core` or `… expanded` — but that's a global setting, not per-repo.

### Phase-to-Change Mapping

### Rules

- Read the active change's `proposal.md` and `design.md` before any implementation
- Do not implement beyond the current change's `tasks.md`
- If design decisions change mid-implementation, update `design.md` first, then continue
- Commit after each completed task group, not at the very end
- One active change at a time — archive before proposing the next
- Archive checklist (definition of done) lives in `.claude/rules/documentation.md`

## TOOL CALL OPTIMIZATION

### Batching Strategy

- Read all relevant source files before generating edits — never edit blind
- Group file writes by module boundary (e.g. all `commands/` changes in one pass)
- Run `cargo check` after each logical change group, not after every single file

### Parallel Execution

- File reads with no dependencies can be parallelized
- `cargo check` and `cargo test` are sequential — always check before test

## DEPLOYMENT & MAINTENANCE

### Release Flow

Releases are fully automated — do not manually tag, bump versions, or edit the changelog.

1. Push conventional commits to `master`
2. Release Please opens/updates a release PR with `CHANGELOG.md` update and `Cargo.toml` version bump
3. Merge the release PR → Release Please pushes a version tag
4. cargo-dist triggers on the tag:
    - Builds all target platforms
    - Publishes `repograph.rb` formula to `maikbasel/homebrew-tap` (shared with changelog-x)
    - Uploads shell + PowerShell installers and tarballs to GitHub Release
    - Publishes to crates.io

### Distribution Channels

| Channel         | Install command                                                                                           |
|-----------------|-----------------------------------------------------------------------------------------------------------|
| crates.io       | `cargo install repograph`                                                                                 |
| Homebrew        | `brew install maikbasel/tap/repograph`                                                                    |
| Shell installer | `curl -LsSf https://github.com/maikbasel/repograph/releases/latest/download/repograph-installer.sh \| sh` |
| PowerShell      | `irm https://github.com/maikbasel/repograph/releases/latest/download/repograph-installer.ps1 \| iex`      |
