## Context

repograph's agent surface is generated, not hand-authored: a single canonical `BODY` + `SUMMARY` in `crates/repograph-core/src/agent_artifact.rs` is spliced by `repograph init` into a per-agent file (Claude `SKILL.md`, Cursor `.mdc`, AGENTS.md / CONVENTIONS.md / Windsurf single delimited block). That body is deliberately **read-only** — it documents only `context/list/status/switch/doctor` and instructs the agent *not* to run mutating commands, because "the registry is the user's to manage."

This change adds a second, **mutation-capable** generated skill (`repograph-setup`) so agents can register repos, group them into workspaces, and update existing entries — without abandoning the architecture (single source of truth, generated by `init`, installed at user scope so it's discoverable from any project). Building it surfaces two CLI shortfalls that must be fixed for a safe agent loop: there is no non-lossy *edit* path, and the mutating commands produce no machine-verifiable output.

Constraints carried in unchanged: exit-code contract, stdout=data / stderr=diagnostics, the delimiter byte-stability contract (only the managed region is ever rewritten), `thiserror` + `tracing`, no `unwrap`/`todo!`, real `git2` + `tempdir` tests.

## Goals / Non-Goals

**Goals:**
- A generated `repograph-setup` skill that drives `add`, `remove`, `workspace …`, and a new `edit`, behind a plan → confirm → execute → verify workflow.
- A `Capability` dimension in the artifact layer so one `init` run can emit both the consumer and setup skills per agent.
- A non-lossy `repograph edit` command, including rename that preserves workspace memberships.
- `--json` confirmation output on the mutating commands so an agent verifies in one call.
- An industry-standard freshness story: version-stamped marker + a read-only `doctor` check that points at `repograph init`.

**Non-Goals:**
- Driving `repograph init` agent-toolchain reconfiguration from the skill.
- Bulk/recursive directory-scan registration.
- Workspace rename / description edit.
- A `doctor --fix` auto-remediation, or auto-rewriting artifacts from `update` (rejected; see Decisions).

## Decisions

### D1 — Distribution: generated by `init`, not hand-authored
The setup skill is a second canonical body emitted through the existing splice machinery, exactly like the consumer skill. **Why over a standalone `SKILL.md`:** keeps a single source of truth (can't drift from the CLI), lands at user scope (discoverable from any project), and reuses the delimiter/idempotency contract. A hand-authored file would drift and wouldn't auto-distribute.

### D2 — A `Capability` enum threaded through the artifact layer
Introduce `enum Capability { Consumer, Setup }`. `resolve_path`, `render_artifact`, and `ArtifactResult` gain a capability dimension; `install_artifacts` emits **one or two** artifacts per agent depending on the agent's file model:
- **Wholly-owned-file agents** (`claude-code`, `cursor`) get a **discrete second file**: `skills/repograph-setup/SKILL.md`, `.cursor/rules/repograph-setup.mdc`. Two independently-triggerable skills.
- **Flat-file agents** (`agents-md`, `aider`, `windsurf`) have one shared file with one managed block and no multi-skill concept; the setup guidance is **inlined** — `render_artifact` concatenates Consumer + Setup body into the single delimited block. Those files are always in context, so separate triggering was never relevant.

The existing `wholly_owned_file(agent)` helper already draws exactly this line. **Why not a discrete second block for flat-file agents:** two delimited regions in one file complicates the splice/idempotency contract for no gain.

### D3 — Confirm-before-write workflow in the setup body
The `SETUP_BODY` instructs the agent to resolve & validate inputs, **show the user the concrete plan** (what will be registered/grouped/edited), run the mutation only on confirmation, then verify via the command's `--json` result. **Why:** honors the "registry is the user's" ethos while still letting the agent act; matches the best-practices "plan → validate → execute → verify" pattern. The consumer body's "ask the user to run `add` themselves" line is rewritten to **delegate to the setup skill** instead of dead-ending.

### D4 — New `edit` command instead of `add --update`
`config.add_repo` hard-rejects duplicate names (`Conflict`, exit 5), so today editing = `remove` + `add`, which is **lossy**: `Workspace.members` is `Vec<String>` of repo names and `remove_repo` does not touch memberships, so the repo silently drops out of every workspace (becomes dangling). New `config.edit_repo` mutates in place; `repograph edit <name> [--name --description --stack --path]` with `--name` **rewrites every `workspace.members` entry** matching the old name. **Why a distinct command over overloading `add`:** clearer intent, distinct exit semantics (`edit` of a missing name → `3 NotFound`; `--name` to an existing name → `5 Conflict`), and it keeps `add`'s "must not already exist" contract intact.

### D5 — `--json` confirmation on mutators (opt-in, contract-preserving)
`add`, `remove`, `workspace create/add/remove/rm`, and `edit` gain a `--json` confirmation envelope on stdout (e.g. `{ "action": "add", "repo": { "name", "path", "stack", … } }`). Without `--json` they stay silent on stdout (confirmation on stderr) — preserving the existing "diagnostics never reach stdout" scenarios. **Why:** an agent's verify step becomes one call instead of a follow-up `list --json`; also aligns the mutators with the project's stdout=data contract, which they currently sidestep by emitting nothing.

### D6 — Freshness = version-stamped marker + read-only `doctor` check (reject auto-update)
Stamp the managed marker with the body version (e.g. `<!-- repograph:begin vN -->`). Add a new read-only `Check` variant to the doctor catalog that, per selected agent × capability, resolves the expected artifact path and reports `warn` when it's **missing** or its stamp is **older than the running binary**, with remediation message "run `repograph init`". **Why this over having `update` rewrite files (option A):** the industry norm is that `doctor`-class commands diagnose and *print* the fix (`brew doctor`, `flutter doctor`), never silently mutate; `repograph update` only fires for installer/tarball installs (Homebrew/cargo bypass it), so auto-rewrite there would miss most users; and `doctor --json` is referenced by the **read-only** consumer skill — making it write files would break the contract that lets agents call it freely. `repograph init` stays the sole regenerator (it already resolves scope). Config keeps persisting only `[agents].selected`, not scope — the doctor check only needs to resolve expected paths and compare a stamp.

## Risks / Trade-offs

- **Existing installs show "stale" until re-init.** After upgrading, every prior install flags `warn` in doctor and the new setup skill is absent until `repograph init` is re-run. → Mitigation: the doctor message states the exact remediation; this is the accepted, standard "explicit regeneration" trade-off (D6).
- **Two skills could mis-trigger** (agent loads consumer when it needed setup or vice-versa). → Mitigation: sharp, non-overlapping `SUMMARY`/`SETUP_SUMMARY` trigger phrasing — consumer = read/resolve, setup = register/group/update; consumer body explicitly hands off to setup for mutation.
- **Agent mutating without genuine consent.** → Mitigation: the confirm-before-write workflow (D3) plus `--json` plan/verify; mutations are local, reversible TOML edits, and `remove`/`workspace rm` leave repos on disk untouched.
- **`edit --name` rename missing a membership rewrite would orphan groupings.** → Mitigation: `edit_repo` owns the rewrite in core (not the command layer), covered by a dedicated `tempdir` test asserting workspace membership survives the rename.
- **Capability dimension ripples through artifact tests.** `init_artifacts.rs` and any test asserting one-artifact-per-agent must learn the two-capability shape. → Mitigation: outside-in tests written first against the new emission contract.

## Resolved deviations (as built)

- **Doctor freshness probes both scopes.** Because install scope is not persisted (kept that way per D6), `with_skill_artifact_check` resolves *both* the user- and project-scope path for each (agent, capability) and treats the artifact as present if found at either — only "missing at both" warns. It is a binary-appended check (`DoctorReport::with_skill_artifact_check`), mirroring the existing `with_index_check` rather than living inside `DoctorReport::run`, since it needs `home`/`cwd` the core `run` doesn't take.
- **Version stamp is an integer baked into `DELIMITER_BEGIN`.** `ARTIFACT_BODY_VERSION: u32 = 1` is the source of truth; the literal `DELIMITER_BEGIN = "<!-- repograph:begin v1 -->"` carries it, kept in sync by a unit test. A version-agnostic `DELIMITER_BEGIN_PREFIX` drives splice detection so an older-version block is rewritten in place, and `installed_version()` parses the stamp for doctor.
- **`edit` description semantics.** `RepoEdit.description` is `Option<Option<String>>`: omitted leaves it unchanged, `--description ""` clears it, `--description <text>` sets it. The command rejects an all-empty flag set with a usage error (exit 2).
- **Doctor test hermeticity.** The freshness check reads `host_home()`, so the `doctor` acceptance helpers now pin `HOME`/`USERPROFILE`/cwd to the fixture tempdir (mirroring `InitFixture`) — otherwise they would read the developer's real `~/.claude`.
